# AI-oriented 开源个人基金研究平台需求书 v0.4

## 1. 文档定位

本文档用于从零建设一个开源、个人可运行、AI-friendly 的基金研究平台，服务个人投资研究、量化实验、基金分析方法学习、AI Agent 调用和开源社区协作。原始参考需求覆盖公募基金信息展示、基金分析算法和系统基础功能；本版将目标升级为“AI-oriented 个人开源研究底座”，重点补充免费/公开数据源策略、可复现算法、结构化研究包、证据链、Tool API、基金画像指纹和轻量技术架构。

本平台的核心价值不在于简单展示基金资料，也不是给传统页面外挂一个聊天框，而在于形成一套人类研究者和 AI Agent 都能使用的可解释、可追踪、可复现基金研究底座。基金、经理、净值、持仓、指标、算法结果、证据和结论都应尽量结构化，使 AI 能通过稳定接口取数、运行分析、引用来源并生成可复核结论。平台不追求机构级数据完整性，而追求在免费公开数据约束下把研究方法、算法框架、数据口径和证据系统做扎实。

v0.4 将建设策略从“一期覆盖完整主动权益研究平台”调整为“先建设可信 AI-ready 研究底座，再分阶段增强算法和产品能力”。第一期不追求把模拟持仓、交易能力和综合评分做成权威结论，而优先建立免费公开数据接入、数据质量基线、证据链、Research Packet、Tool API 和可复核的基础分析闭环。后续阶段再逐步引入模拟持仓、动态归因、评分体系、相似基金、Agent 化研究和更多资产类型。

## 2. 建设目标

### 2.1 总体目标

建设一个面向个人研究者、开源贡献者、基金爱好者、量化学习者和未来 AI Agent 的基金研究工具，覆盖公募基金标的从信息查询、净值分析、公开持仓分析、算法评分、组合模拟、结构化研究包到证据化研究结论的个人研究流程。

### 2.2 业务目标

1. 提升个人研究效率：将基金基本信息、净值表现、公开持仓、基金经理、基金公司、公告报告和因子暴露集中到一个本地工具。
2. 提升分析深度：在免费公开数据可得范围内，对主动权益基金、债券基金、指数基金/ETF 建立不同算法框架。
3. 提升研究可复现性：统一指标口径、算法版本、数据快照和评分规则，让研究结果可以复算、复查、复盘。
4. 构建 AI-friendly 研究底座：输出标准化 Research Packet、Evidence、Fund Fingerprint 和 Tool API，让 AI 能直接读取、组合和解释平台数据。
5. 支持插件化数据源：默认接入免费/公开数据，预留用户自行配置商业数据源或本地数据文件的能力。
6. 支持开源协作：算法、数据适配器、指标定义、页面组件、研究模板和 Agent 工具定义均可由社区贡献。

### 2.3 非目标

1. 平台不提供交易下单、荐股荐基、自动投资或收益承诺能力。
2. 平台不作为正式投资顾问工具，仅用于个人研究、学习和方法验证。
3. 平台不以实时高频交易为目标，主要服务日频、季度、半年度、年度研究分析。
4. 平台不随项目分发第三方原始数据，避免违反数据源授权或网站条款。
5. 平台一期不追求复杂研报管理、在线协作和完整文档知识库；公告/报告主要作为可解析证据来源，而不是做传统文档管理系统。

## 3. 目标用户画像

个人开源版不按企业系统的岗位和权限划分用户角色，而按使用动机划分目标用户画像。固收分析、本地配置、数据源管理等不单独定义为用户角色，而作为平台功能模块提供。

| 用户画像 | 优先级 | 典型诉求 | 重点功能 |
| --- | --- | --- | --- |
| 个人基金研究者 | 核心 | 分析单只基金、筛选候选基金、沉淀研究结论 | 基金详情、收益风险、公开持仓、风格暴露、模拟持仓、收益拆解、综合评分、研究包 |
| 量化/算法学习者 | 核心 | 学习因子、回归、归因、组合优化、评分和回测方法 | 算法实验、披露期回测、指标解释、参数配置、Notebook 示例、Tool API |
| ETF/指数研究者 | 次核心 | 对比 ETF/指数基金，构建低成本指数替代组合 | 指数基金优选、ETF 组合构建、跟踪误差分析、费率和流动性对比 |
| AI Agent/自动化使用者 | 核心 | 通过接口取数、运行分析、生成可引用结论 | Research Packet、Evidence API、指标注册表、研究任务模板、结构化输出 |
| 开源贡献者 | 扩展 | 扩展数据源、改进算法、提交可视化组件和文档 | 数据适配器、算法模块、测试样例、插件机制、贡献指南 |

## 4. 平台范围

平台功能按六层划分：

1. 数据层：基金、净值、持仓、股票、债券、指数、行业、报告、公告、市场因子、公司和经理维度数据。
2. 语义层：统一基金 ID、证券 ID、基金经理 ID、指标字典、实体关系、时间快照和数据来源等级。
3. 算法层：收益风险指标、风格板块暴露、模拟持仓、收益拆解、交易能力、综合评分、债基因子暴露、ETF 复制和反选基金。
4. 证据层：将每个结论绑定输入数据、算法版本、参数、日期区间、报告/公告片段、置信度和误差。
5. API/Tool 层：面向前端、Notebook 和 AI Agent 的结构化接口，提供取数、筛选、运行算法、解释评分和生成研究包能力。
6. 产品层：基金详情、筛选检索、分析页面、研究包、看板、基金池、组合工具和轻量导出。

### 5.2 数据源策略

1. 免费公开数据优先：默认接入 AKShare、证监会基金信息披露/XBRL、基金公司公告、上交所/深交所基金公告、东方财富/天天基金公开页面等个人研究常用来源。
2. 不分发第三方原始数据：开源仓库只提供数据适配器、字段定义、解析逻辑和示例小样本，不随项目发布批量抓取后的原始数据。
3. 数据源插件化：用户可自行配置 Tushare、Wind、Choice、iFinD、本地 CSV/Parquet、SQLite/DuckDB 等数据源；商业数据源只作为可选插件，不作为默认依赖。
4. 数据来源留痕：所有外部数据接入需保留来源、接口、拉取时间、交易日、字段版本和修正状态。
5. 分层可靠性标记：官方披露数据标为 A 级，开源接口聚合数据标为 B 级，网页解析数据标为 C 级；算法结果应展示所依赖数据源等级。
6. 对免费数据不足的功能不删除，而是降低承诺：通过估计、近似、置信度、残差和人工校验提示来保留模拟持仓、收益拆解、交易能力、债基因子等核心研究功能。
7. AI 友好输出：关键数据和算法结果需提供稳定 JSON Schema，避免只存在于前端图表或自由文本中。
8. 文本数据轻量化：一期不做复杂研报展示和全文文档管理，只做公告/报告解析、片段索引、来源定位和结构化抽取。

### 5.3 数据质量规则

1. 净值连续性：缺失、异常跳变、复权错误、分红拆分不一致需报警。
2. 持仓完整性：持仓比例合计、现金缺失、行业缺失、证券代码失效需标记。
3. 口径一致性：基金分类、行业分类、风格分类、债券评级口径需版本化。
4. 数据回溯：历史数据重算时必须保留原快照，避免覆盖历史研究结论。
5. 算法输入校验：任何模型计算前需输出输入覆盖率、缺失率、异常值数量。
6. AI 输出校验：提供给 AI 的 Research Packet 必须包含数据日期、缺失字段、数据源等级、算法版本和不适用说明。
7. 证据完整性：任何自动生成的结论若缺少证据或置信度，应标记为“待复核”，不能进入默认研究结论。

### 5.4 AI 友好数据规范

1. 所有核心实体使用稳定 ID：基金、基金经理、基金公司、股票、债券、指数、行业、报告片段均需有唯一标识。
2. 所有时间序列数据保留交易日、更新时间、数据源、复权口径和缺失状态。
3. 所有指标统一注册：字段名、中文名、计算公式、输入数据、适用范围、更新频率、解释文本、局限性。
4. 所有算法结果统一附带 metadata：算法名称、版本、参数、输入数据快照、运行时间、置信度和错误信息。
5. 研究包采用可版本化 JSON Schema，前端、Notebook、导出和 AI Agent 共用同一份结构化数据。
6. 文本片段只存必要索引和引用信息：报告标题、页码/章节、原文片段、抽取字段、来源链接或本地路径。

### 5.5 结论可信度与估计隔离规则

平台必须区分“披露事实”“确定性计算结果”“估计结果”和“研究观察”。任何自动生成内容都不能只因为形式上可计算，就进入默认研究结论。

1. 结论可信度门禁：每条结论进入默认研究包前，必须通过数据完整度、数据源等级、算法适用范围、残差阈值、证据完整度和更新时间检查。
2. 结论状态分级：`fact` 表示公开披露事实，`computed` 表示基于确定输入的规则计算，`estimated` 表示模型估计，`observation` 表示研究观察，`needs_review` 表示待复核。
3. 估计污染隔离：模拟持仓、动态归因、交易能力等估计类结果必须统一使用 `estimated_*` 字段，默认不得直接参与综合评分、基金排名或高置信度结论。
4. 置信度不足降级：若输入覆盖率、拟合误差、披露滞后、数据源等级或模型适用性不达标，系统应自动降级为观察或待复核，而不是输出确定性判断。
5. 残差优先展示：收益拆解、风格暴露和模拟持仓应显式展示未解释残差；当残差较高时，应优先提示“无法解释”，不得强行归因。
6. Research Packet diff：同一基金不同日期的研究包应支持差异对比，展示经理、规模、净值指标、公开持仓、风格暴露、风险提示、证据和结论状态的变化。
7. AI 引用约束：AI 只能引用带 evidence ID、metadata 和结论状态的内容；对 `estimated` 和 `needs_review` 内容必须保留不确定性表述。

## 6. 功能需求

### 6.1 基金信息类

#### 6.1.1 基金基本信息

展示基金基础信息，至少包括：

1. 基金代码、简称、全称、基金公司、托管行、成立日、到期日。
2. 基金一级分类、二级分类、投资类型、运作方式、是否 ETF/联接/指增/QDII/FOF。
3. 最新规模、历史规模变化、份额变化。
4. 管理费、托管费、销售服务费、认购/申购/赎回费率。
5. 是否加入自选、是否重点关注、是否加入黑名单、是否标记为风险关注。
6. 当前状态：正常、清盘、暂停申购、暂停赎回、限购、转型。

#### 6.1.2 基金经理概况

展示基金经理维度信息：

1. 当前基金经理、任职日期、任职天数、离任经理。
2. 在管基金数量、在管基金规模、权益/固收/混合/指数产品规模分布。
3. 历史管理基金清单、任职区间、任职收益、同类排名。
4. 基金经理风格标签：成长/价值、大盘/小盘、行业偏好、换手偏好、集中度偏好。
5. 经理变更提醒：新任、共管、离任、团队变化。

#### 6.1.3 基金组合特征

支持按报告期查看组合特征：

1. 基金规模、股票仓位、债券仓位、现金仓位、转债仓位、其他资产仓位。
2. 前十大股票仓位、前五大债券仓位、前三行业仓位。
3. 行业分布、市值分布、风格分布、集中度指标。
4. 与上一报告期相比的仓位变化、行业变化、重仓股变化。
5. 与同类基金均值、基准指数、基金经理历史均值对比。

#### 6.1.4 基金历史收益风险特征

支持多时间区间和自定义区间：

1. 今年以来、近 1 月、近 3 月、近 6 月、近 1 年、近 3 年、近 5 年、成立以来、基金经理任职以来。
2. 区间收益、年化收益、同类排名、分位数。
3. 最大回撤、年化波动率、下行波动率、VaR、CVaR。
4. 夏普比率、卡玛比率、索提诺比率、信息比率、跟踪误差。
5. 胜率、月度正收益比例、回撤修复天数。
6. 指标必须展示计算口径、基准选择、交易日范围。

#### 6.1.5 基金净值

1. 展示单位净值、累计净值、复权净值曲线。
2. 支持与业绩比较基准、同类均值、沪深 300、中证 500、中证 800、中债指数等对比。
3. 支持查看回撤曲线、滚动收益、滚动波动、滚动夏普。
4. 支持定投模拟、持有期收益分布、买入时点胜率。

#### 6.1.6 持仓具体信息

1. 股票持仓：代码、名称、行业、市值、持仓市值、持仓比例、估值指标、涨跌幅。
2. 债券持仓：代码、名称、债券类型、评级、久期、票息、到期日、持仓比例。
3. 转债持仓：代码、名称、正股、转股溢价率、纯债溢价率、持仓比例。
4. 支持按报告期切换、按资产类型筛选、按行业/评级聚合。
5. 支持持仓变动分析：新增、增持、减持、退出。

#### 6.1.7 持有人信息

1. 展示个人投资者、机构投资者、内部员工持有比例。
2. 展示历史变化趋势。
3. 提示单一机构占比高、机构快速赎回、内部员工持有异常变化等风险。

### 6.2 基金分析类

#### 6.2.1 风格板块特征

适用范围：主动权益基金、偏股混合基金、灵活配置中权益仓位较高基金。

算法目标：计算基金在市值、风格、行业/板块上的历史暴露，并以日频或近似日频更新。

输入数据：

1. 基金日度净值收益。
2. 市值风格指数、行业指数、主题指数、宽基指数收益。
3. 基金定期持仓、模拟持仓结果。
4. 股票市值、行业、估值、成长、质量、动量等因子数据。

核心方法：

1. 基于净值收益的滚动回归暴露：使用带约束回归、岭回归、Lasso 或贝叶斯收缩估计基金对风格/行业指数的暴露。
2. 基于持仓的真实暴露：按披露持仓权重汇总市值、风格、行业暴露。
3. 混合估计：在报告期披露附近提高持仓权重，在非披露期使用净值回归修正。
4. 暴露平滑：使用指数加权或状态空间模型降低噪声。

输出结果：

1. 大盘/中盘/小盘暴露曲线。
2. 成长/价值/均衡暴露曲线。
3. 行业/板块暴露热力图。
4. 风格漂移指标、风格稳定性评分、行业集中度评分。
5. 与基金合同、基金经理历史风格、同类均值的偏离提示。

验收标准：

1. 日度计算任务可稳定运行。
2. 暴露合计约束合理，不能长期出现明显不可解释负暴露或超高杠杆暴露。
3. 至少对历史典型基金进行人工验证，结果应符合研究员认知。

#### 6.2.2 模拟持仓

适用范围：主动权益基金。

算法目标：在只披露季度/半年度持仓的情况下，估计基金在非披露区间的股票、转债持仓结构。该模块属于高不确定性估计能力，不得被表达为真实持仓，不得在未通过回测阈值前进入默认综合评分或高置信度研究结论。

阶段定位：

1. 一期：不作为默认核心结论，只提供模拟持仓可行性实验、披露期回测、拟合误差、置信度和失败样本分析。
2. 二期：在通过披露期回测阈值后，作为主动权益分析页的可选估计视图。
3. 三期以后：仅当模型稳定性、基金类型覆盖和错误提示机制成熟后，才允许作为动态归因、交易能力和评分体系的输入之一。

输入数据：

1. 报告期披露持仓。
2. 基金日度净值收益。
3. 股票/转债日度收益、成交量、停复牌、涨跌停信息。
4. 基金合同约束、仓位区间、历史换手率。

核心方法：

1. 约束优化：在给定候选股票池内，最小化模拟组合收益与基金净值收益的误差。
2. 稀疏约束：控制持仓数量和权重集中度，避免过拟合。
3. 换手约束：限制相邻窗口权重变化，符合基金交易成本和披露持仓逻辑。
4. 候选池构建：由上期持仓、同经理偏好股、行业龙头、定报新增股、风格相近股、转债池组成。
5. 多模型融合：披露持仓锚定 + 净值拟合 + 风格暴露约束 + 行业约束。

输出结果：

1. 模拟持仓明细：证券代码、名称、模拟权重、置信度、来源标记。
2. 模拟组合净值拟合误差、日度残差、跟踪误差。
3. 股票/转债/现金仓位估计。
4. 模拟持仓变化趋势。
5. 失败原因、低置信度原因和不可解释残差。

验收标准：

1. 对已披露期做回测，模拟持仓与真实披露持仓的行业分布、重仓股召回率、收益拟合误差达到约定阈值。
2. 输出必须包含置信度、拟合误差、输入覆盖率、候选池来源和结论状态，不能将模拟结果表现为真实持仓。
3. 低置信度样本必须自动降级为 `needs_review`，不得进入默认研究结论。
4. 支持研究员手动锁定/排除特定证券后重算。
5. 一期验收重点是“知道何时不可用”，而不是追求所有基金都能给出看似精确的模拟持仓。

#### 6.2.3 收益拆解分析

适用范围：主动权益基金。

算法目标：将基金季度收益拆分为 beta、配置、板块轮动、选股、转债、打新、隐形收益等来源。

拆解维度：

1. Beta 收益：来自市场整体暴露。
2. 配置收益：来自股票/债券/现金/转债等大类资产配置。
3. 板块轮动收益：来自行业或板块权重相对基准的偏离。
4. 个股选择收益：来自板块内选股效果。
5. 转债收益：来自转债仓位和转债个券表现。
6. 打新收益：来自新股中签和上市收益。
7. 隐形收益：净值收益无法由披露或模拟持仓解释的部分，需单独标记。

核心方法：

1. 静态归因：使用报告期初或报告期披露持仓固定权重进行 Brinson 类归因，作为一期默认口径。
2. 动态归因：使用模拟持仓或滚动估计权重进行日度归因汇总，二期以后作为估计口径引入。
3. 板块级归因：按行业/板块拆分配置收益和选股收益。
4. 个股级归因：对重仓股、新增股、退出股计算贡献。
5. 残差分析：将无法解释收益拆分为交易残差、未披露持仓、费用、现金拖累和模型误差。

输出结果：

1. 季度收益拆解瀑布图。
2. 历史收益来源堆叠图。
3. 板块轮动收益明细。
4. 选股收益明细。
5. 隐形收益占比和异常提示。

验收标准：

1. 各拆解项合计应与基金实际收益接近，误差需在页面显式展示。
2. 所有归因结论必须可追溯到基准、报告期、算法版本和输入持仓。
3. 一期不得把动态归因残差解释为确定交易能力；残差较高时应输出“无法解释”或“低置信度观察”。

#### 6.2.4 交易能力分析

适用范围：主动权益基金，优先支持披露全部持仓或半年度持仓较完整的基金。

算法目标：估计基金通过调仓交易创造或损失的收益，并识别基金经理交易能力。

核心方法：

1. 基于相邻报告期持仓变化估算交易区间。
2. 对新增、增持、减持、退出证券计算交易收益贡献。
3. 使用价格路径区间构建保守、中性、乐观三种交易假设。
4. 对交易收益按行业、个股、买入/卖出行为拆分。
5. 与基金经理历史交易收益、同类平均交易收益对比。

输出结果：

1. 报告期交易收益对净值贡献。
2. 个股交易收益明细。
3. 买入能力、卖出能力、止盈止损能力、换手效率指标。
4. 交易能力稳定性评分。

验收标准：

1. 页面必须提示交易时间不可完全观测，结论为估计结果。
2. 不同交易假设下结果差异较大时需提示低置信度。

#### 6.2.5 综合得分及排名

适用范围：主动权益基金。

算法目标：形成可解释、可配置、可回测的基金综合评分体系。

阶段定位：一期只做指标分项展示和筛选排序，不做权威综合评分。综合评分基础版应放在二期，并且必须先完成评分回测、权重解释、缺失值惩罚、估计结果隔离和版本管理。

评分维度建议：

1. 收益能力：长期年化收益、超额收益、胜率、持有期收益分布。
2. 风险控制：最大回撤、波动率、下行风险、回撤修复速度。
3. Alpha 能力：选股收益、行业配置收益、超额收益稳定性。
4. 交易能力：交易收益贡献、换手效率、买卖时点能力。
5. 风格稳定性：风格漂移、合同偏离、行业集中度。
6. 规模适配：规模过小风险、规模过大带来的容量压力。
7. 团队稳定性：基金经理任职稳定、基金公司投研支持、共管变化。
8. 持有人稳定性：机构集中度、申赎压力、内部员工持有。

评分方法：

1. 指标标准化：按基金类别、风格、规模分组做分位数或 z-score 标准化。
2. 权重配置：支持平台默认权重、研究员自定义权重、策略模板权重。
3. 稳健处理：极值缩尾、缺失值惩罚、样本期不足降权。
4. 回测验证：评分必须能回测不同分位基金组合的未来收益、回撤和换手。
5. 解释输出：展示总分、分项得分、排名变化和扣分原因。

验收标准：

1. 任一基金分数均可解释到指标层。
2. 评分版本必须保存，历史评分不可因新版本算法被静默覆盖。
3. 筛选结果支持按风格、板块、行业、基金公司、基金经理、规模分组。
4. 任何依赖 `estimated_*` 的评分必须显著标注估计来源和置信度；默认综合评分不得直接依赖未通过回测阈值的模拟持仓、交易能力或动态归因结果。

#### 6.2.6 基金公司画像频谱

适用范围：主动权益基金经理和基金公司。

算法目标：在指定时间维度内计算基金经理/基金公司相较市场的 beta 和 alpha，并结合管理规模形成画像。

核心方法：

1. 对基金经理管理基金合成经理组合。
2. 对基金公司旗下主动权益基金合成公司组合。
3. 使用市场基准和风格因子回归估计 beta、alpha。
4. 结合规模、产品数量、行业偏好、风格稳定性生成频谱图。

输出结果：

1. 基金公司 alpha-beta 气泡图，气泡大小代表管理规模。
2. 基金经理 alpha-beta 分布图。
3. 公司风格频谱：成长/价值、市值、行业、集中度。
4. 规模扩张与业绩变化趋势。

#### 6.2.7 债基因子暴露

适用范围：纯债基金、一级债基、二级债基、可转债基金、短债基金。

算法目标：计算债券基金在票息、利率、曲线、信用、转债和 alpha 等因子上的暴露。

因子建议：

1. 票息因子。
2. 利率波动因子。
3. 曲线斜率因子。
4. 曲线凸度因子。
5. 隐含 AAA 信用因子。
6. 隐含 AA 信用因子。
7. 信用下沉因子。
8. 久期因子。
9. 杠杆因子。
10. 转债因子。
11. 流动性因子。
12. Alpha/残差因子。

核心方法：

1. 基于净值收益对债券因子收益序列做滚动回归。
2. 基于持仓债券估算真实久期、评级、期限、票息和转债暴露。
3. 对二级债基额外纳入权益 beta 和转债 beta。
4. 使用状态空间模型或 Kalman Filter 平滑日度暴露。

输出结果：

1. 因子暴露曲线。
2. 因子收益贡献拆解。
3. 久期、信用、杠杆、转债风险雷达图。
4. 与同类债基、基准和基金经理历史均值对比。

验收标准：

1. 因子暴露方向需符合已知产品策略。
2. 对短债、纯债、二级债、转债基金需使用不同模板，避免单一模型误判。

#### 6.2.8 指数基金分析与优选

适用范围：ETF、ETF 联接、普通指数基金、指数增强基金。

评价维度：

1. 规模、流动性、成交额、持有人结构。
2. 管理费率、托管费率、综合费率。
3. 跟踪误差、年化跟踪误差、日均偏离、最大偏离。
4. 超额收益和稳定性，指增产品需单独看 alpha。
5. 申赎效率、折溢价、上市状态、成分复制质量。

输出结果：

1. 同指数产品对比表。
2. 综合优选评分。
3. 跟踪误差和偏离曲线。
4. 指增 alpha 和风险暴露分析。

#### 6.2.9 ETF 组合构建

算法目标：给定目标指数或目标指数权重组合，使用现有 ETF 构建低成本、低跟踪误差、满足约束的替代组合。

用户输入：

1. 目标指数或指数组合及权重。
2. 可选 ETF 池。
3. 单只 ETF 最小/最大权重。
4. ETF 规模、流动性、费率、跟踪误差限制。
5. 组合数量上限、换手限制、行业偏离限制。

核心方法：

1. 二次规划：最小化组合与目标指数收益序列的跟踪误差。
2. 约束优化：规模、流动性、费率、权重上下限、数量上限。
3. 稳健优化：避免历史样本过拟合。
4. 再平衡模拟：月度/季度再平衡，输出换手和成本。

输出结果：

1. 推荐 ETF 组合权重。
2. 历史拟合收益、跟踪误差、最大偏离。
3. 组合费率、规模、流动性、换手成本。
4. 与目标指数行业/风格偏离。

#### 6.2.10 标的股票反选基金

算法目标：输入一篮子股票代码，按真实持仓和模拟持仓计算基金对该股票池的暴露比例，并排序展示。

输入：

1. 股票代码列表。
2. 可选基金范围。
3. 统计口径：披露持仓、模拟持仓、二者加权。
4. 时间范围：最新报告期、近一年平均、指定日期。

输出：

1. 基金名称、基金代码、基金经理、基金公司。
2. 股票池持仓比例、个股贡献比例。
3. 模拟持仓置信度。
4. 同类排名、历史变化。

扩展场景：

1. 输入行业/主题股票池，筛出主题暴露高的主动基金。
2. 输入禁投股票池，筛查潜在组合风险。
3. 输入自选股票池，寻找间接配置工具。

### 6.3 AI-oriented 系统功能

#### 6.3.1 基金检索及筛选

1. 支持基金代码、简称、基金经理、基金公司、主题、行业、持仓股票关键词检索。
2. 支持按基金类别、规模、成立年限、经理任职年限、费率、托管行、申赎状态筛选。
3. 一期支持按收益、风险、回撤、规模、经理任职年限、数据完整度排序；综合评分、alpha、风格稳定性、交易能力排序作为后续能力。
4. 支持保存筛选条件为研究模板。
5. 支持基金池管理：关注池、白名单、黑名单、待调研池、组合候选池。

#### 6.3.2 Research Packet 基金研究包

研究包是平台的核心亮点之一，用于把单只基金的研究上下文打包成 AI 和人都能读取的结构化材料。

1. 支持为单只基金生成标准化 JSON 研究包。
2. 支持生成 Markdown 摘要，方便用户阅读、复制、提交 issue 或放入 Notebook。
3. 一期研究包至少包含：基金基础信息、基金经理、净值指标、公开持仓、持仓变化、风格/行业暴露、静态归因、残差、风险提示、关键证据、结论状态和数据质量摘要。
4. 后续研究包可逐步加入：模拟持仓、动态收益拆解、交易能力、综合评分、相似基金、基金画像指纹、债基风险、ETF 对比和组合穿透。
5. 研究包必须包含 metadata：数据日期、数据源等级、算法版本、缺失字段、置信度、免责声明。
6. 支持按主题生成不同视角研究包：单基金体检、经理画像、风格漂移、持仓穿透、债基风险、ETF 对比。

#### 6.3.3 Evidence 证据链

证据链用于保证 AI 和算法结论可追溯、可复核。

1. 每条自动结论都应绑定至少一条证据记录。
2. 证据类型包括：原始数据字段、时间序列指标、算法结果、报告/公告片段、图表数据、用户手工备注。
3. 证据记录包含：证据 ID、实体 ID、来源、日期区间、算法版本、参数、置信度、原文片段或数据摘要。
4. 页面和 API 均能返回证据列表，AI 生成回答时可引用证据 ID。
5. 低置信度、缺失关键数据或残差过高的结论必须标记为“待复核”。

#### 6.3.4 AI Tool API

平台应预留一组稳定、轻量、结构化的工具接口。MVP 可先作为本地 REST API 或 Python 函数暴露，未来可包装成 MCP server、OpenAPI 或 Agent tools。

一期最小接口：

1. `get_fund_profile(fund_code)`：获取基金基础信息、经理、分类、规模、费率。
2. `get_nav_metrics(fund_code, start, end)`：获取收益、回撤、波动、夏普、卡玛等净值指标。
3. `get_disclosed_holdings(fund_code, report_date)`：获取公开披露持仓。
4. `run_exposure_analysis(fund_code, params)`：运行或读取风格/行业暴露、静态归因和残差。
5. `build_research_packet(fund_code, template)`：生成研究包。

后续扩展接口：

1. `screen_funds(filters)`：按结构化条件筛选基金。
2. `compare_funds(fund_codes)`：对比多只基金的关键指标和画像指纹。
3. `run_simulated_holding(fund_code, params)`：运行或读取模拟持仓实验结果。
4. `run_return_attribution(fund_code, params)`：运行或读取动态收益拆解。
5. `explain_score(fund_code, score_version)`：返回总分、分项得分、扣分原因和证据。
6. `find_similar_funds(fund_code, metric_space)`：查找相似基金。
7. `find_anomalies(scope, rules)`：发现风格漂移、评分跳变、低置信度高分等异常。
8. `diff_research_packets(fund_code, left_snapshot, right_snapshot)`：对比两个研究包快照。

接口要求：

1. 返回 JSON，不依赖前端页面解析。
2. 每个返回结果包含 `data`、`metadata`、`evidence`、`warnings` 和结论状态。
3. 对不适用的基金类型返回明确原因，而不是空结果。
4. 支持 dry-run 或 explain 模式，方便 AI 先判断能否调用。
5. 对 `estimated` 和 `needs_review` 内容必须保留不确定性标记。

#### 6.3.5 Fund Fingerprint 基金画像指纹

基金画像指纹用于相似基金搜索、聚类、异常发现和 AI 上下文压缩。

1. 为每只基金生成标准化特征向量。
2. 特征维度包括：收益风险、回撤、波动、风格暴露、行业暴露、集中度、换手估计、alpha、交易能力、债基因子、ETF 跟踪指标等。
3. 特征按基金类型使用不同模板，避免主动权益、债基、指数基金混用同一指纹。
4. 支持相似基金检索：同风格、同经理风格、同持仓、同风险收益、同因子暴露。
5. 支持异常识别：名义分类和实际暴露不一致、风格漂移、规模变化和业绩异常、评分高但置信度低。

#### 6.3.6 研究任务模板

研究任务模板用于把研究流程固化为可复现步骤，也方便未来 AI Agent 自动执行。

建议模板：

1. 单基金体检：基础信息、净值、持仓、风格、收益拆解、评分、风险提示。
2. 基金经理画像：历史产品、任职收益、风格稳定性、交易能力、规模变化。
3. 主动权益筛选：筛选条件、评分排序、风格分组、候选池输出。
4. 风格漂移监控：暴露变化、合同偏离、同类偏离、证据列表。
5. ETF 替代组合：目标指数、候选 ETF、约束、拟合误差、组合权重。
6. 股票池反选基金：输入股票池、披露持仓暴露、模拟持仓暴露、置信度。
7. 债基风险扫描：久期、信用、转债、杠杆、收益残差、同类对比。

每个模板应定义输入、步骤、调用接口、输出字段、适用基金类型和失败处理。

#### 6.3.7 轻量公告解析

研报展示和完整文档管理不是平台亮点，一期不作为重点建设。保留的文本能力只服务于结构化抽取和证据引用。

1. 支持解析基金定期报告、招募说明书、基金合同、重要公告。
2. 抽取字段包括：持仓、基金经理观点、投资策略、风险提示、业绩比较基准、费用、限制条款。
3. 支持保存报告片段引用，不要求建设复杂 PDF 阅读器、全文批注和文档权限系统。
4. 支持把公告片段作为 Evidence 绑定到研究包和 AI 输出。

#### 6.3.8 研究看板

建议新增：

1. 今日重点变化：净值异动、排名大幅变化、基金经理变更、规模异常变化。
2. 基金池监控：关注基金收益、回撤、风格漂移、持仓风险。
3. 算法预警：模拟持仓低置信度、归因残差过高、评分异常跳变。
4. AI 预警：研究包缺关键字段、结论缺证据、评分高但置信度低、数据源等级过低。
5. 市场环境：指数表现、行业轮动、风格因子、债券曲线、信用利差。

#### 6.3.9 组合与基金池分析

建议新增：

1. 支持用户创建基金组合，输入基金权重。
2. 计算组合收益、回撤、波动、风格暴露、行业暴露、基金经理集中度。
3. 基于基金模拟持仓穿透到底层股票/债券，分析重仓重叠度。
4. 支持基金组合优化：控制回撤、相关性、风格偏离和流动性风险。
5. 支持输出组合研究包，供 AI 对组合风险、持仓重叠和风格偏离进行解释。

#### 6.3.10 结构化导出

传统 Excel/PDF 导出不是亮点，保留为基础能力，但优先级低于结构化研究包和 API。

1. 优先支持 JSON、JSONL、CSV、Markdown 导出。
2. 支持导出 Research Packet、Evidence、Fund Fingerprint、筛选结果和算法结果。
3. 可选支持 Excel；PDF 报告可作为后续能力，不作为 MVP 必需项。
4. 导出内容必须包含数据日期、算法版本、数据源等级、置信度和免责声明。

## 7. 算法平台需求

### 7.1 算法资产管理

1. 每个算法需定义名称、版本、适用基金类型、输入字段、输出字段、参数和维护说明。
2. 支持算法离线回测、版本切换和结果复算。
3. 任一页面展示的算法结果需可追溯到算法版本和运行批次。
4. 支持算法说明文档和口径说明页面。

### 7.2 模型实验与回测

1. 支持不同参数、不同样本期、不同基金范围的实验管理。
2. 支持记录指标、图表、输出文件和实验备注。
3. 支持训练集/验证集/样本外测试。
4. 支持评分模型的未来收益分层回测。
5. 支持模拟持仓算法的披露期回测。

### 7.3 可解释性

1. 综合评分要解释到分项指标。
2. 模拟持仓要展示置信度和拟合误差。
3. 收益拆解要展示残差。
4. 债基因子暴露要展示因子定义和回归稳定性。
5. ETF 组合构建要展示约束是否生效、为何选择某只 ETF。

### 7.4 指标注册表

指标注册表是 AI-oriented 架构的基础组件。

1. 每个指标必须注册字段名、中文名、英文名、所属实体、数据类型、单位、计算公式。
2. 每个指标必须说明输入数据、适用基金类型、更新频率、缺失值处理、极值处理和局限性。
3. 每个指标必须提供人类可读解释和 AI 可读 schema。
4. 指标变更需记录版本，历史研究包应能追溯当时使用的指标定义。

### 7.5 结论置信度系统

平台应为算法结论和 AI 结论提供统一置信度，不追求伪精确。

置信度由以下因素综合形成：

1. 数据完整度：输入数据覆盖率、缺失率、异常值比例。
2. 数据源等级：官方披露、开源聚合、网页解析、本地用户数据。
3. 算法稳定性：回测误差、拟合残差、参数敏感性、窗口稳定性。
4. 样本期长度：基金成立时间、经理任职时间、可用净值长度。
5. 适用性：基金类型是否适配当前算法模板。
6. 证据充分性：结论是否有足够指标、数据或公告片段支撑。

输出等级建议：

1. 高：数据完整、来源可靠、模型稳定、证据充分。
2. 中：数据基本完整，但存在估计或局部缺失。
3. 低：数据缺失较多、残差较高、模型仅能给方向性参考。
4. 待复核：证据不足或模型不适用，不应自动生成确定性结论。

### 7.6 Agent 运行约束

如果未来接入 AI Agent，默认遵守以下约束：

1. Agent 只能调用已注册 Tool API，不能直接读取未登记的本地敏感文件。
2. Agent 输出必须引用 evidence ID 或明确说明缺少证据。
3. Agent 不能把模拟持仓、交易能力估计、收益拆解估计表述为事实。
4. Agent 生成研究结论时必须附带数据日期、算法版本、置信度和不适用说明。
5. Agent 不能自动给出买卖建议，只能输出研究观察、风险提示和可复核分析路径。

## 8. 技术架构建议

### 8.1 推荐架构

1. 前端：Web 平台，支持基金详情、图表、筛选、研究包、证据查看和任务运行。
2. 后端：API 服务 + Tool API + 算法服务 + 数据任务服务分层。
3. 数据存储：默认使用 SQLite/DuckDB + 本地文件缓存；数据量扩大后可选 PostgreSQL、ClickHouse 或对象存储。
4. 语义层：维护实体 ID、指标注册表、数据源等级、时间快照和 schema。
5. 证据层：存储 Evidence、算法 metadata、报告片段索引、结论置信度。
6. 任务编排：按日、季度、半年度运行数据更新和算法计算，也支持按需生成研究包。
7. 算法执行：Python 为主，支持批量计算、实验管理、结果落库。
8. 本地优先：默认单用户本地运行；多用户、权限和审计作为可选扩展，不作为 MVP 必需项。
9. AI 接入：MVP 不依赖大模型，但所有核心数据和算法结果都通过 JSON Schema 和 Tool API 暴露，未来可包装成 MCP server 或 OpenAPI。

### 8.3 自研重点

以下模块建议作为平台核心价值，不宜简单套模板：

1. 主动权益基金模拟持仓算法。
2. 主动权益收益拆解和隐形收益解释。
3. 基金经理交易能力分析。
4. 综合评分体系和未来收益验证框架。
5. 债基因子体系和不同债基类型模板。
6. 基金公司画像频谱和经理画像标签体系。
7. 基于中国公募基金数据口径的数据质量和归因校验体系。
8. Research Packet、Evidence、Fund Fingerprint 和 Tool API 的统一 schema。
9. 相似基金搜索、风格漂移识别和低置信度高分异常发现。

## 9. 开源与数据合规

1. 项目定位为个人研究工具，页面和导出文件需展示免责声明：算法结果仅用于个人研究，不构成投资建议。
2. 仓库不分发批量第三方原始数据，只分发代码、字段定义、解析逻辑、测试样例和少量脱敏/示例数据。
3. 数据适配器需在文档中说明来源、用途、频率、已知限制和可能的接口失效风险。
4. 对网页解析类数据需尊重源站 robots、服务条款和访问频率，默认设置限速、缓存和重试。
5. 用户自行配置商业数据源时，平台只负责本地读取，不负责授权；导出时需保留数据源标记。
6. 模拟持仓、交易能力、收益拆解等估计结果必须显著标注置信度、误差和局限。

## 10. 非功能需求

### 10.1 性能

1. 基金检索响应时间：常规条件下小于 2 秒。
2. 单只基金详情页核心信息加载：小于 3 秒。
3. 复杂算法图表可异步加载，页面需展示任务状态。
4. 日频算法任务应在每日数据到齐后可配置窗口内完成。

### 10.2 可用性

1. 本地启动流程应简单，目标是普通个人电脑可运行。
2. 算法任务失败需在页面或日志中提示，并支持重跑。
3. 数据源异常需切换备用数据或标记缺失，不得静默生成错误结论。

### 10.3 可维护性

1. 指标口径配置化。
2. 算法参数配置化。
3. 数据任务和算法任务日志可检索。
4. 支持按基金、日期、算法版本重算。

### 10.4 本地安全与隐私

1. 默认单用户本地运行，不强制登录。
2. 用户自定义数据源密钥、Cookie、Token 等必须保存在本地配置或环境变量中，不能提交到仓库。
3. 个人研究笔记、自选基金池、个人组合和导出文件默认只保存在本地。
4. 如未来支持多用户部署，再补充登录、权限、审计和访问控制。

## 11. 验收指标

### 11.1 产品验收

1. 可完成单只基金从基础信息、经理、净值、公开持仓、收益风险到算法分析的完整查看。
2. 可通过筛选条件找到目标基金，并保存筛选模板。
3. 可生成单只基金 Research Packet，包含数据日期、算法版本、数据源等级、缺失字段、置信度和关键证据。
4. 可查看自动结论对应的 Evidence 列表。
5. 可通过结构化导出获得 JSON/Markdown/CSV，而不是只依赖页面截图或 Excel。

### 11.2 算法验收

一期算法验收以确定性指标、公开持仓分析、风格/行业暴露、静态归因和残差展示为主。模拟持仓、动态归因、交易能力、综合评分、基金画像指纹和 ETF 组合属于后续阶段算法验收项，不作为一期上线门槛。

1. 风格板块暴露：对典型基金的风格识别与研究者判断基本一致。
2. 模拟持仓：回测披露期时，行业暴露、重仓股召回率、收益拟合误差达到评审阈值。
3. 收益拆解：拆解项合计与实际收益误差可控，残差可解释。
4. 交易能力：输出三种交易假设和置信度，避免单一确定性结论。
5. 综合评分：分层回测能展示评分与未来收益/风险的关系。
6. 基金画像指纹：能支持相似基金检索，并能解释相似原因。
7. ETF 组合：目标跟踪误差、约束满足率、换手成本可量化。

### 11.3 AI-ready 验收

1. 核心实体和指标具备稳定 JSON Schema。
2. Tool API 能返回 `data`、`metadata`、`evidence`、`warnings`。
3. AI 或 Notebook 可不依赖前端页面，直接调用接口完成单基金体检。
4. 所有自动结论均有证据 ID 或待复核标记。
5. 模拟持仓、收益拆解、交易能力等估计类结果不会被接口描述为事实。

### 11.4 数据验收

1. 关键基金数据覆盖率达到 MVP 约定阈值。
2. 净值、持仓、规模、基金经理、费率等字段通过抽样校验。
3. 数据修正和算法重算有记录。
4. 研究包能展示缺失字段和数据源等级。

## 12. 分期建设建议

平台长期目标仍然是建设强大、好用、可扩展的 AI-oriented 基金研究平台，但建设节奏必须从“可信底座”开始。每个阶段都应形成可运行、可验证、可导出的闭环，而不是把所有功能都堆到一期。

### 12.0 第零阶段：数据可得性与口径试验

目标：先验证免费公开数据能支撑哪些结论，避免后续算法建立在不稳定输入上。

范围：

1. 盘点 AKShare、官方披露、基金公司公告、交易所公告、本地 CSV/Parquet 的可得字段。
2. 对 20-50 只典型主动权益基金建立样本集，覆盖大规模、小规模、高换手、低换手、行业主题、均衡风格等类型。
3. 建立数据源等级、字段覆盖率、缺失率、更新时间、异常值和快照机制。
4. 定义基金实体、净值、公开持仓、指标注册表、Evidence、Research Packet、Tool API 返回结构的首版 schema。
5. 输出数据质量基线报告，明确哪些结论可做、哪些只能估计、哪些暂时不做。

### 12.1 一期：可信 AI-ready MVP

目标：先让个人研究者和 Notebook/AI 工具都能用免费公开数据完成“单只主动权益基金体检 + 结构化研究包 + 证据链”的闭环。

范围：

1. 基金基础信息、基金经理、净值、收益风险、公开披露持仓、规模、费率、持有人结构。
2. 免费数据接入只保留最小可靠集合：官方披露/公告、本地文件导入、AKShare 便捷源；暂不维护多套商业数据源适配器。
3. 数据质量页：数据源等级、字段覆盖率、缺失字段、更新时间、异常提示、快照记录。
4. 基础主动权益分析：收益风险指标、公开持仓分析、行业/风格暴露、静态归因、残差展示。
5. 结论可信度门禁、估计污染隔离、Evidence、Research Packet、Research Packet diff、结构化导出。
6. 一期 Tool API 最小集合：`get_fund_profile`、`get_nav_metrics`、`get_disclosed_holdings`、`run_exposure_analysis`、`build_research_packet`。
7. 一期不做默认综合评分，不把模拟持仓作为默认结论，不做交易能力分析，不做股票池反选基金。

### 12.2 二期：算法验证与受控估计

目标：在已有可信底座上，逐步引入高不确定性算法，但先验证、标注和隔离，再参与产品结论。

范围：

1. 模拟持仓实验模块：候选池构建、约束优化、披露期回测、失败样本分析、置信度与残差展示。
2. 收益拆解增强：在静态归因基础上引入估计动态归因，但必须使用 `estimated_*` 字段并保留残差。
3. 综合评分基础版：只使用确定性指标和通过回测的估计指标，展示分项得分、权重、版本和回测表现。
4. 算法实验管理：参数版本、样本集、回测指标、模型适用范围、失败案例库。
5. 研究员手动校验能力：锁定/排除证券、调整基准、标记低置信度结论。

### 12.3 三期：发现能力与研究工作台

目标：从单基金体检扩展到基金池研究、对比、筛选和异常发现，形成日常可用的研究工作台。

范围：

1. Fund Fingerprint 基金画像指纹。
2. 相似基金搜索和基金对比。
3. 风格漂移、名义分类偏离、低置信度高分、集中度异常、持有人结构异常等发现能力。
4. 基金池/自选池管理、候选池筛选、跟踪提醒。
5. 标的股票反选基金，优先使用公开披露持仓，模拟持仓只作为低置信度补充。
6. 研究任务模板：单基金体检、经理画像、风格漂移、持仓变化、风险扫描。

### 12.4 四期：ETF/指数、组合与更多资产类型

目标：扩展到指数基金、ETF、基金组合和粗粒度债基研究，增强平台实用性。

范围：

1. 指数基金分析与优选、ETF 组合构建、跟踪误差、费率和流动性对比。
2. 基金组合穿透分析、持仓重叠、风格偏离、回撤和相关性分析。
3. 债基因子暴露粗粒度版本：久期、信用、杠杆、转债、利率风险、流动性风险。
4. 组合 Research Packet 和组合 Evidence。
5. 基金公司画像频谱和基金经理团队画像。

### 12.5 五期：Agent 化研究与开源生态

目标：让 AI Agent 能安全调用平台能力，自动执行研究任务、生成可引用初稿，并支持社区扩展。

范围：

1. MCP/OpenAPI tools，围绕已注册 Tool API 暴露能力。
2. Agent 研究任务编排：自动取数、运行分析、生成草稿、列出证据、标注待复核点。
3. 公告片段语义检索和摘要，支持基金经理观点、风险提示、投资策略抽取。
4. Notebook 示例、社区研究模板、插件贡献指南。
5. 可选数据源插件市场，但不把商业数据源作为默认依赖。

## 16. MVP 验收场景

1. 个人研究者搜索一只主动权益基金，查看基础信息、经理、净值、历史收益风险、公开持仓、规模、费率和持有人结构。
2. 用户进入主动权益基础分析页，查看公开持仓行业分布、风格/行业暴露、静态归因和未解释残差。
3. 用户查看数据质量页，知道该基金哪些字段来自官方披露、哪些来自 AKShare、哪些缺失或需要复核。
4. 用户生成该基金 Research Packet，看到关键观察、证据 ID、数据源等级、算法版本、结论状态、缺失字段、置信度和免责声明。
5. 用户对比同一基金两个日期的 Research Packet diff，看到经理、规模、持仓、风险指标、风格暴露和证据变化。
6. Notebook 或本地脚本调用 5 个 Tool API，完成同一只基金的单基金体检，并返回 JSON。
7. 用户按近三年收益、最大回撤、基金规模、基金经理任职年限、数据完整度筛选基金，保存为自选候选池。
8. 用户导出研究包 Markdown/JSON/CSV，文件包含算法版本、数据日期、数据来源等级、Evidence、warnings 和结论状态。

## 17. 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| 数据源字段不完整 | 算法无法稳定计算 | 第零阶段先做字段盘点和数据质量基线，缺字段时自动降级 |
| 免费接口失效或限流 | 数据无法更新 | 官方披露和本地文件作为 A 级主路径，AKShare 作为 B 级便捷源，强制缓存和快照 |
| 个人维护多个适配器成本过高 | 长期维护不可持续 | 一期只维护最小可靠数据源，其他数据源只保留插件协议 |
| 模拟持仓过拟合 | 研究结论误导并污染下游模块 | 一期降级为实验模块，二期才受控引入；使用 `estimated_*` 隔离，不直接进入默认评分 |
| 收益归因残差过高 | 解释力不足 | 拆出残差，展示误差，不强行解释，不把残差包装成交易能力 |
| 综合评分过早权威化 | 用户误以为平台能判断基金优劣 | 一期不做默认综合评分；二期必须先做评分回测、权重解释和估计隔离 |
| AI 幻觉或过度解释 | 误导用户 | 强制 evidence ID、结论状态、置信度、待复核标记和不构成投资建议提示 |
| Tool API schema 频繁变化 | 后续 Agent 难接入 | 从 MVP 开始版本化 schema，保留兼容策略 |
| 不同基金类型混用模型 | 结果失真 | 按主动权益、债基、指数、FOF 建立模型模板，未适配类型直接标记不适用 |
| 评分黑盒化 | 用户不信任 | 分项指标、权重、版本、回测全部可见 |
| 第三方数据授权限制 | 开源分发有风险 | 仓库不分发原始数据，只分发适配器、schema、解析逻辑和示例数据 |
| 算法计算耗时 | 页面体验差 | 异步任务、结果缓存、增量计算 |
| 范围膨胀 | 个人项目无法完成 | 每期只验收一个闭环，未通过前不进入下一期 |

## 18. 下一步建议

1. 先执行第零阶段：选取 20-50 只典型主动权益基金，做免费公开数据字段盘点和数据质量基线。
2. 先定义一期 schema：基金实体、净值指标、公开持仓、指标注册表、Evidence、Research Packet、Tool API 返回结构、结论状态。
3. 产出一期原型图：基金检索、基金详情、主动权益基础分析、研究包、证据链、数据质量、Tool API 调试。
4. 产出一期技术方案：本地存储、数据快照、数据源等级、任务日志、schema 版本、结构化导出。
5. 明确一期不做项：默认综合评分、模拟持仓正式结论、交易能力分析、股票池反选基金、Agent 自动研究。
6. 在一期跑通后，再启动二期算法评审：重点评审模拟持仓、动态归因、综合评分在免费数据约束下的适用范围、回测阈值和失败样本。
