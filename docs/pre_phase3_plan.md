# Pre-Phase 3 补充开发计划

> 对照 v0.4 需求书 §12 分期建设建议，覆盖 Phase 1 遗留项补全、Phase 2 收尾、前端架构重构三大板块。
> 目标：为 Phase 3（发现能力与研究工作台）建立完整、可信、可用的底座。

---

## 0. 总览

### 0.1 当前状态速览

| 阶段 | 验收状态 | 核心问题 |
|------|----------|----------|
| Phase 1 (§12.1) | ✅ 基本达标 | 4 项缺口：本地文件适配器、结构化导出、基金池持久化、Evidence/API 调试页 |
| Phase 2 (§12.2) | ⚠️ 框架完成 | 3 项收尾：交易能力模块缺失、CVXPY 路径未激活、评分维度不全 |
| 前端 | ⚠️ 功能可用，视觉粗糙 | 980 行单文件 CSS、无设计系统、无统一组件库、暗色导航+浅色内容割裂 |

### 0.2 本计划范围

```
[P2.5] Phase 1 补全 (4项)  ──┐
                              ├── 并行 ──┐
[P2.6] Phase 2 收尾 (3项)  ──┘          │
                                        ├──→ [Phase 3]
[P2.7] 前端架构重构           ──────────┘
```

### 0.3 优先级原则

1. **阻塞性优先**：基金池持久化是 Phase 3 硬依赖，必须先做。
2. **数据先于算法**：股票日行情数据覆盖是 CVXPY 和评分维度的共同前提。
3. **前端重构先行于 Phase 3 功能开发**：Phase 3 需要全新的 UI 模式（多基金对比/看板/异常发现），旧前端无法支撑。
4. **可并行**：Phase 1/2 后端补全与前端架构重构可并行推进。

## 2. Phase 2 收尾（P2.6）

### P2.6-1 交易能力分析模块 ★v0.4 二期明确要求

**需求来源**：v0.4 §6.2.4 交易能力分析 + §12.2 二期范围。

**现状**：完全缺失。无 `trading_ability.py` 模块、无 API、无前端页面。

**实现计划**：

1. **算法模块**（`src/fund_research/analysis/trading_ability.py`）：
   ```
   输入：相邻报告期持仓变化 + 区间价格路径
   核心方法：
   - 识别新增/增持/减持/退出证券
   - 对每笔交易构建保守/中性/乐观三种假设收益区间
   - 按行业、个股、买卖方向拆分交易收益
   - 计算买入能力/卖出能力/换手效率指标
   输出：
   - 报告期交易收益贡献（含三种假设区间）
   - 个股交易收益明细
   - 交易能力稳定性评分
   - 结论状态：estimated（v0.4 明确要求标注估计性质）
   ```

2. **ORM 表**：`trading_ability_result`（fund_code, report_date, scenario, contribution, detail_json, confidence, conclusion_status, algorithm_version）

3. **API**（v2_router）：
   - `POST /analysis/trading-ability` — 运行交易能力分析
   - `GET /analysis/trading-ability/{fund_code}` — 读取已存结果

4. **验收标准**（v0.4 §6.2.4）：
   - 页面必须提示交易时间不可完全观测，结论为估计结果
   - 不同交易假设下结果差异较大时需提示低置信度
   - 所有结果使用 `estimated_*` 字段隔离

### P2.6-2 模拟持仓 CVXPY 优化路径激活

**需求来源**：v0.4 §6.2.2 模拟持仓验收标准 + §12.2。

**现状**：[simulated_holding.py](file:///e:/Vibe/fund-research/src/fund_research/analysis/simulated_holding.py) 已实现 CVXPY + 朴素复制双路径，但因股票日行情数据覆盖不足，默认 fallback 到朴素复制。

**实现计划**：

1. **数据覆盖**：
   - 扩展 `update` CLI：批量更新样本基金的持仓股票日行情（至少覆盖近 2 年）
   - 优先覆盖 30 只 P2B 样本基金的持仓股票池
   - 确保股票日行情表 `stock_daily` 有足够数据触发 CVXPY 路径

2. **路径激活验证**：
   - 在 P2B 验证框架中增加检查项：CVXPY 路径触发率
   - 目标：对有足够股票数据的基金，CVXPY 路径触发率 > 80%
   - 回测阈值验证：行业相关性 > 0.7，top10 召回 > 50%（v0.4 §6.2.2 验收标准）

3. **失败样本分析**：记录无法触发 CVXPY 的基金及原因（数据不足/停牌过多/候选池过小）

### P2.6-3 综合评分维度补全

**需求来源**：v0.4 §6.2.5 综合得分 8 维度 + §12.2。

**现状**：[scoring_dimensions.py](file:///e:/Vibe/fund-research/src/fund_research/analysis/scoring_dimensions.py) 已实现 6 个维度函数，但仅收益+风险 2 维度有真实可靠数据，其余依赖 estimated 或未验证输入。

**实现计划**：

1. **Alpha 能力维度**（`compute_alpha`）：
   - 当前使用代理基准。需要接入真实市场基准指数收益（沪深300/中证500/中证800）
   - 计算 Jensen's Alpha + 超额收益稳定性
   - 数据来源：AKShare 指数日行情（`fetch_index_daily` 已实现）

2. **风格稳定性维度**（`compute_style_stability`）：
   - 基于已完成的风格暴露时间序列，计算滚动窗口风格漂移度
   - 数据已就绪（Phase 1 风格暴露结果），仅需完善计算逻辑

3. **规模适配维度**（`compute_scale`）：
   - 基于基金规模历史，计算规模过大/过小惩罚分
   - 数据已就绪（`fund_scale` 表），仅需完善评分逻辑

4. **团队稳定性维度**（`compute_team`）：
   - 基金经理任职年限 + 公司投研支持评分
   - 数据已就绪（`fund_manager_tenure` 表），需完善评分逻辑

5. **持有人稳定性维度**（`compute_holder`）：
   - 机构集中度 + 申赎压力
   - 当前 `holder_structure` 数据为全市场汇总（AKShare 限制），需评估是否能支撑单基金级评分
   - 若数据不可用，标记为 `needs_review` 并降权

6. **评分回测完善**：
   - 在维度补全后重跑 P2B 回测
   - 验证 IC 单调性、分层收益区分度
   - 目标：至少 5/8 维度有真实数据支撑，默认 `allow_estimated=False` 时可算出有意义分数

### P2.6-4 动态归因真实基准数据（可选，可延至 Phase 3）

**需求来源**：v0.4 §6.2.3 收益拆解。

**现状**：[dynamic_attribution.py](file:///e:/Vibe/fund-research/src/fund_research/analysis/dynamic_attribution.py) 使用代理基准（×0.9）和代理行业收益。这是数据源工程问题，不是算法问题。

## 3. 前端架构重构（P2.7）

### 3.0 现有前端交互问题诊断

在讨论重构方案前，先明确当前前端「不像基金研究系统」的具体问题。以下不是视觉问题，而是信息架构和交互逻辑问题：

#### 问题 1：首页是空的——没有工作台入口

当前 [HomePage.tsx](file:///e:/Vibe/fund-research/frontend/src/pages/HomePage.tsx) 只有一个标题、一个后端状态指示灯、两个 QuickLink 卡片。一个基金研究系统的首页应该是**研究工作台**——用户一进来就应该看到：最近关注的基金动态、待复核的结论、数据更新状态、快捷操作。而不是一个「关于本平台」的介绍页。

#### 问题 2：基金筛选页缺乏研究工具感

当前 [FundListPage.tsx](file:///e:/Vibe/fund-research/frontend/src/pages/FundListPage.tsx) 的问题：
- 筛选条件只有 5 个简单下拉/输入框平铺一行，没有分组、没有高级筛选
- 表格只有 7 列（代码/名称/类型/规模/经理/评审状态/操作），**没有展示任何收益/风险指标**——筛选结果无法直接比较
- 排序和筛选是分离的（筛选在一行，排序在另一行），交互割裂
- 搜索后需手动点「搜索」按钮，没有输入即筛选
- 没有列选择、没有批量操作、没有「加入基金池」
- 没有保存筛选条件功能

#### 问题 3：基金详情页子导航是平铺链接，没有层次

当前 [FundDetailPage.tsx](file:///e:/Vibe/fund-research/frontend/src/pages/FundDetailPage.tsx) 在标题下方平铺了 8 个子链接（持仓/暴露与归因/评分/模拟持仓/动态归因/研究包/对比/校验），没有分组、没有层次。用户无法理解这些功能之间的关系：
- 哪些是「基础分析」？哪些是「实验性分析」？
- 「暴露与归因」和「动态归因」有什么区别？
- 「研究包」和「对比」是分析功能还是输出功能？

专业系统应该用**分组 Tab** 或**侧边面板**来组织这些功能，比如：
- 概览（基础信息+净值指标）
- 持仓分析（公开持仓+持仓变动）
- 风格与归因（风格暴露+静态归因+动态归因[实验]）
- 评分与实验（综合评分+模拟持仓[实验]+交易能力[实验]）
- 研究输出（研究包+对比+证据链）
- 研究员校验

#### 问题 4：研究包页面是 JSON dump，不是研究文档

当前 [ResearchPacketPage.tsx](file:///e:/Vibe/fund-research/frontend/src/pages/ResearchPacketPage.tsx) 把研究包的每个 section 用 `<details>` 折叠，里面直接 `JSON.stringify` 输出原始 JSON。这完全不可读——研究包应该是**结构化的研究文档**，有标题、有指标卡片、有图表、有表格、有证据引用，而不是 raw JSON viewer。

#### 问题 5：页面间无上下文衔接

- 从基金列表点「详情」进入详情页，看不到筛选时的上下文
- 从详情页进入子页面（如持仓），子页面只显示 `← 返回基金详情`，没有面包屑导航
- 页面间没有状态传递（如选中的时间区间、报告期），每个页面独立重新加载

#### 问题 6：实验管理和验收报告脱离主流程

[ExperimentsPage.tsx](file:///e:/Vibe/fund-research/frontend/src/pages/ExperimentsPage.tsx) 和 [P2BValidationPage.tsx](file:///e:/Vibe/fund-research/frontend/src/pages/P2BValidationPage.tsx) 在顶部导航中作为独立入口，但它们本质是**算法验证工作流**的一部分，应该和对应的算法页面（模拟持仓/评分/动态归因）关联，而不是孤立的管理页面。

#### 问题 7：数据质量页是静态说明，不是监控面板

当前 [DataQualityPage.tsx](file:///e:/Vibe/fund-research/frontend/src/pages/DataQualityPage.tsx) 只展示了数据源等级说明表和一句「请运行 check-data」的提示。这应该是**数据质量监控面板**——展示各数据源的覆盖率、缺失率、最近更新时间、异常快照，而不是一个静态说明页。

#### 问题 8：导航结构扁平，无法体现系统层次

当前 NavBar 只有 5 个一级链接（首页/基金筛选/实验管理/验收报告/数据质量），所有功能平铺。一个基金研究系统至少需要两层导航：
- **全局功能**（基金筛选/基金池/数据质量/实验管理）
- **单基金上下文**（进入某基金后，该基金的分析功能子导航）

### 3.1 重构设计理念

**Visual thesis**：一个安静的投研工作台——纸面般的暖白底色、墨级排版、单一琥珀色标记重要信息；界面像研究员的桌面，而非仪表盘。

**设计方向**：Editorial Research Desk —— Financial Times 的研究气质 × Linear 的工具克制 × Bloomberg 的数据密度。

**核心原则**：
- 冷静的层级关系，不靠卡片堆叠
- 强排版和间距系统，不靠装饰
- 少色彩，一个强调色
- 信息密集但可读
- 极简 chrome，卡片只在卡片本身就是交互时才用

**差异化记忆点**：暖纸色底 + 琥珀色强调 + 衬线标题——不是又一个蓝白 SaaS 模板，而是一个有研究气质的专业工具。

#### 3.2.2 字体系统

放弃 Inter（最典型的 AI slop 字体），采用衬线标题 + 无衬线正文 + 等宽数字的组合，呼应研究出版物的排版传统。

```css

  /* 展示字体：Fraunces —— 有性格的衬线，用于标题和品牌 */
  --font-display: 'Fraunces', 'Noto Serif SC', Georgia, serif;

  /* 正文字体：IBM Plex Sans —— 中性但不平庸，可读性极强 */
  --font-body: 'IBM Plex Sans', 'Noto Sans SC', -apple-system, sans-serif;

  /* 等宽字体：IBM Plex Mono —— 数字对齐，金融数据必备 */
  --font-mono: 'IBM Plex Mono', 'JetBrains Mono', 'SF Mono', monospace;
}
```

**字体使用规则**：
- 所有标题（h1-h4）、品牌名、页面标题用 `--font-display`
- 正文、标签、按钮、导航用 `--font-body`
- **所有数字**（净值、收益率、指标值、百分比）用 `--font-mono` + `font-variant-numeric: tabular-nums`
- 中文自动 fallback 到 Noto Sans/Serif SC

### 3.5 动效设计

克制但有存在感，共 3 个动效层次：

1. **页面入场**：staggered fade-up（150ms 间隔，300ms duration），内容区模块依次淡入上移
2. **路由切换**：内容区淡出(100ms) → 淡入(200ms)，无滑动
3. **交互微动效**：
   - 表格行 hover：背景色过渡 120ms
   - Tab 切换：下划线滑动 200ms ease-out
   - 按钮 hover：背景色过渡 100ms
   - Modal/Drawer：fade + scale 200ms

使用 CSS transition 为主，不引入额外动画库（当前项目无 framer-motion 依赖，保持轻量）。

#### 3.9.8 全局交互模式变更

以下交互模式贯穿所有页面：

1. **面包屑导航**：所有子页面顶部有面包屑（`基金筛选 > 110011 > 持仓分析`），替代当前的 `← 返回基金详情`
2. **全局时间区间**：基金详情页的时间区间选择器是全局的，所有 Tab 内图表共用
3. **状态保持**：基金详情页切换 Tab 时不重新请求基础信息，只加载 Tab 对应数据
4. **骨架屏加载**：所有数据加载用骨架屏，非 spinner
5. **空状态引导**：每个空状态都有明确的引导文案和操作按钮
6. **错误重试**：所有错误状态都有重试按钮
7. **键盘快捷键**：`/` 聚焦搜索框，`Esc` 关闭弹层
8. **响应式**：最小支持 1024px 宽度，移动端暂不优化（专业工具桌面优先）
9. **数字格式化**：所有数字用 tabular-nums 等宽对齐，百分比/小数位数统一
10. **状态标签**：所有 conclusion_status 用统一的 StatusBadge 组件，颜色语义一致
