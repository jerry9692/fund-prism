# 第零阶段：免费数据可复现性与披露口径验证

## 0.1 阶段定位

第零阶段是平台建设的前置验证阶段。目标不是做产品功能，而是在写任何算法代码之前，系统性地回答一个核心问题：

> **在免费公开数据约束下，我们能可靠地支撑哪些研究结论？哪些只能估计？哪些暂时做不了？**

整个平台的算法设计、分期节奏、置信度系统都建立在这个试验的结果之上。

第零阶段不把 AKShare 当作默认主数据源，而是把 AKShare 作为 B 级便捷源的验证对象。对于净值、持仓、公告这三个核心数据域，**必须至少验证一条不依赖 AKShare 的 A 级官方披露路径**（如证监会 XBRL、基金公司官网公告 PDF、交易所披露页面），确保平台不因单一接口库失效而全线瘫痪。

## 0.2 阶段目标

| 编号 | 目标 | 验收形式 |
|------|------|----------|
| D0-1 | 盘点免费数据源的实际可得字段 | 字段清单 CSV |
| D0-2 | 建立原始字段到标准字段的映射 | `config/field_mapping_v0.1.yaml` |
| D0-3 | 建立样本基金集并拉取全量数据 | 本地 DuckDB 数据库 |
| D0-4 | 输出字段覆盖率矩阵 | 覆盖率矩阵 Markdown 表格 |
| D0-5 | 评估每个算法模块的数据基础 | 算法可行性评估表 |
| D0-6 | 定义首版核心 schema | Pydantic schema + JSON Schema 导出 |
| D0-7 | 输出数据质量基线报告 | 数据质量基线 Markdown 文档 |

## 0.3 非目标

- 不做前端页面
- 不跑任何分析算法（风格暴露、归因等）
- 不优化数据拉取性能
- 不处理增量更新和定时任务
- 不写正式的数据适配器代码（只写拉取实验脚本）

## 0.4 样本基金选择

### 0.4.1 数量和构成

选取 **30 只** 主动权益基金作为样本集，覆盖以下维度：

| 维度 | 分类 | 目标数量 | 说明 |
|------|------|----------|------|
| 规模 | 大盘（>100亿） | 5 | 代表头部基金 |
| 规模 | 中盘（10-100亿） | 10 | 代表主流产品 |
| 规模 | 小盘（<10亿） | 5 | 代表小型基金 |
| 风格 | 成长 | 5 | 科技、新能源等 |
| 风格 | 价值 | 5 | 红利、金融等 |
| 风格 | 均衡 | 5 | 无明显风格偏好的混合型 |
| 换手 | 预期高换手 | 5 | 基于基金经理历史风格预估，拉取持仓后根据相邻季度持仓变化确认 |
| 换手 | 预期低换手 | 5 | 同上，拉取后确认 |
| 特殊 | 行业主题（医药/消费/科技） | 5 | 代表主题基金 |
| 特殊 | 灵活配置型 | 5 | 仓位变动范围大 |

注：一只基金可同时属于多个维度（如"大规模 + 成长 + 低换手"），总样本 30 只即可。换手率标签先按"预期"标注，第零阶段拉取持仓数据后，基于相邻季度持仓变化率修正，并记录换手率估算来源。

### 0.4.2 选择原则

1. 优先选择成立满 3 年的基金（确保有足够样本期做后续分析）
2. 优先选择有知名度的基金（易方达、华夏、广发、中欧、富国、兴全、交银等头部公司产品）
3. 尽量包含不同基金经理风格的基金
4. 包含至少 2 只已清盘或转类型的基金（验证数据源对中间状态基金的支持）

### 0.4.3 样本集管理

样本集需要版本化，避免后续样本漂移。在 `data/samples/` 下维护：

## 0.5 免费数据可复现性与披露口径验证

### 0.5.1 数据源优先级

按 v0.4 定义的四级数据源等级：

| 等级 | 来源 | 具体渠道 | 第零阶段验证重点 |
|------|------|----------|-----------------|
| A | 官方披露 | 基金公司公告（定期报告 PDF/XBRL）、证监会基金信息披露、上交所/深交所公告 | 验证至少一条净值、持仓、公告的 A 级路径可达 |
| B | 开源接口 | AKShare（便捷源）、Tushare（备用源） | 验证可用性和便利性，但不得作为唯一依赖 |
| C | 网页解析 | 天天基金、东方财富基金页面 | 记录可选路径，评估稳定性和合规风险 |
| LOCAL | 本地文件 | 用户自备 CSV/Parquet | 验证本地文件导入通道 |

**核心原则**：对于净值、持仓、公告三个关键数据域，第零阶段必须在 AKShare 之外，至少验证一条 A 级官方披露路径的实际可达性（不要求完美覆盖，但要确认"如果 AKShare 挂掉，我们还有路可走"）。

## 0.6 数据质量基线

### 0.6.1 检查维度

对每只样本基金的每个数据域，记录以下质量指标：

| 指标 | 含义 | 计算方式 |
|------|------|----------|
| 字段覆盖率 | 非空字段占比 | `non_null_count / total_records` |
| 缺失模式 | 哪些字段经常一起缺失 | 缺失字段共现矩阵 |
| 异常值比例 | 超出合理区间的值 | 人工设定阈值 + 统计检测 |
| 时间连续性 | 交易日序列是否完整 | 交易日历比对 |
| 口径一致性 | 同一基金不同来源的数据是否一致 | 交叉校验 |
| 更新延迟 | 最新可用数据距今天数 | `today - max(trade_date)` |
| 历史回溯长度 | 最早到最新的数据跨度 | `max(date) - min(date)` |

### 0.6.2 重点检查项

1. **净值连续性**（需求书 5.3.1）：
   - 是否存在连续两天以上缺失
   - 日收益率是否出现 >20% 的异常跳变
   - 累计净值和复权净值是否因分红拆分出现不一致

2. **持仓披露粒度**（需求书 5.3.2）：
   - **区分披露层级**：季度报告通常只披露前十大重仓股，半年报和年报才披露全部股票持仓。
     对每只样本基金的每个报告期，记录披露层级：
     - `top10_only`：仅前十大重仓股
     - `full_stock_holdings`：全部股票持仓明细
     - `asset_allocation_only`：仅大类资产配置比例
   - 同一只基金不同报告期的披露粒度可能不同，需逐一标注
   - 股票代码可映射性：持仓中的股票代码是否能与 stock_main 表的代码体系匹配
   - 行业字段来源：是直接来源于报告披露还是事后映射

3. **分类口径一致性**（需求书 5.3.3）：
   - 同一只基金在不同数据源中的分类是否一致
   - 行业分类标准（申万 vs 中信 vs GICS）是否明确标注

### 0.6.3 输出物

1. `docs/phase0/data-quality-baseline.md`：总体数据质量评分、30 只样本基金字段完整度排名、已知问题和规避建议
2. `docs/phase0/disclosure-granularity.md`：每只样本基金每个报告期的披露粒度记录，季度/半年度/年度披露差异分析
3. `docs/phase0/source-risk-register.md`：每个数据源的失效风险、限流风险、字段变化历史、替代路径

## 0.7 算法可行性评估

### 0.7.1 评估框架

对每个计划中的算法模块，基于第零阶段的数据盘点结果，回答：

| 问题 | 说明 |
|------|------|
| 必需的输入字段是否可得？ | 列出每个输入字段的覆盖率 |
| 披露粒度是否满足算法要求？ | 仅前十大 vs 全部持仓 |
| 数据频率是否满足算法要求？ | 日频/季度/半年度 |
| 样本量是否足够？ | 最小要求的时间长度和截面数量 |
| 缺失字段是否有合理的替代方案？ | 降级策略 |
| 如果不可得，采用估计还是不做？ | 决策 |

### 0.7.2 评估结论分级

| 等级 | 含义 | 后续动作 |
|------|------|----------|
| 可做 | 所有必需字段覆盖率 >95%，样本量足够 | 直接进入一期算法开发 |
| 降级可做 | 核心字段可得，部分字段有 10-30% 缺失 | 开发时加入缺失值处理和置信度降级 |
| 仅估计 | 关键字段缺失 >30% 或频率不满足 | 用 `estimated_*` 标记，不做默认结论 |
| 暂不可做 | 必需字段缺失 >50% | 推迟到后续阶段或更换数据源 |

### 0.7.3 待评估的算法模块

| 算法 | 依赖的关键数据 | 披露粒度要求 | 一期/后续 |
|------|---------------|-------------|-----------|
| 收益风险指标 | 日度净值 | 不适用 | 一期 |
| 公开持仓分析 | 季度持仓 | 需要区分前十大/全部持仓 | 一期 |
| 风格/行业暴露（净值回归） | 日度净值 + 风格指数日收益 | 不适用 | 一期 |
| 静态 Brinson 归因 | 季度持仓 + 基准指数 | 至少需要前十大 + 行业分布 | 一期 |
| 模拟持仓 | 季度持仓 + 日度净值 + 股票日行情 | 全部持仓效果更好，前十大也可做 | 二期 |
| 动态收益拆解 | 模拟持仓 + 日度净值 | 依赖模拟持仓质量 | 二期 |
| 交易能力分析 | 相邻季度持仓变化 + 价格路径 | 需要前后两个报告期的持仓 | 二期 |
| 综合评分 | 一期所有指标 + 基准 | 不适用 | 二期 |
| 债基因子暴露 | 债基净值 + 债券因子收益序列 | 需要债券持仓明细 | 四期 |
| ETF 对比与组合 | ETF 日行情 + 成分 + 费率 | 不适用 | 四期 |

### 0.8.2 原始字段到标准字段映射

**这是第零阶段最重要的基础设施交付物之一**。必须产出一份映射表 `config/field_mapping_v0.1.yaml`，定义每个数据源的原始字段名到平台标准字段名的对应关系：

```yaml
# 示例：AKShare 基金净值接口 → 平台标准字段
fund_nav:
  source: akshare.fund_open_fund_info_em
  akshare_version: "1.18.64"
  underlying_source: 天天基金（东方财富）
  underlying_source_level: B
  fields:
    - raw_name: "净值日期"
      canonical_name: trade_date
      data_type: date
      unit: null
      required: true
      transform: "pd.to_datetime"
    - raw_name: "单位净值"
      canonical_name: unit_nav
      data_type: float
      unit: CNY
      required: true
    - raw_name: "日增长率"
      canonical_name: daily_return
      data_type: float
      unit: ratio
      required: true
      note: "AKShare 返回的是百分比数值（如 1.23 表示 1.23%），需除以 100 转为小数"
```

每个数据源的映射需记录：`raw_name`、`canonical_name`、`data_type`、`unit`、`required`、`transform`、`missing_handling`、`note`。

### 0.8.3 Schema 版本策略

- 首版 schema 版本号为 `0.1.0`
- 每次字段变更记录在 `config/schema_changelog.yaml`
- 已有代码中的 `schemas.py` 需要与第零阶段盘点的实际字段对齐

### 0.9.2 可复现要求

第零阶段最怕"今天能拉，明天复现不了"。所有实验必须满足：

1. **环境记录**：每个 Notebook 开头记录 `akshare.__version__`、`sys.version`、拉取日期、操作系统
2. **数据快照**：每次拉取结果写入 DuckDB 时，附带 `fetch_timestamp`、`akshare_version`、`pull_date` 字段
3. **数据 hash**：关键数据集计算 `pandas.util.hash_pandas_object` 或 `hashlib` 摘要，记录在快照日志中
4. **失败日志**：拉取失败的基金、接口、错误信息、重试次数、最终状态记录在 `data/samples/fetch_failures.csv`
5. **样本版本**：`sample_funds_v0.1.csv` 一经确定不得修改；如有样本调整，创建 `sample_funds_v0.2.csv` 并在 changelog 中说明原因
6. **限速**：所有拉取请求间隔 >= 2 秒，避免触发反爬

### 0.9.3 合规要求

1. **不分发原始批量数据**：Notebook 只包含代码和少量示例输出，不附带完整的拉取数据。完整数据仅保存在本地 DuckDB 中，不提交到 Git 仓库
2. **不绕过网站限制**：不使用代理池、不伪造 User-Agent、不突破登录墙
3. **记录底层来源**：所有通过 AKShare 获取的数据需标注底层来源（如天天基金、东方财富），因为实际数据来自这些网站
4. **仅本地研究用途**：拉取的数据仅供个人研究，不用于商业目的

## 0.12 退出标准

第零阶段完成后，必须满足以下条件才能进入一期：

1. [ ] 样本基金核心数据拉取成功率 >= 90%（净值、持仓、经理、规模四个数据域分别统计）
2. [ ] 失败样本已记录失败原因、替代路径和是否影响一期范围的判断
3. [ ] 净值、持仓、公告三个核心数据域，至少各存在一条可用的 A 级或 B 级数据路径（不要求全覆盖，但不能是单一 AKShare 接口）
4. [ ] 原始字段到标准字段的映射表已产出，一期可用的接口字段已确定
5. [ ] 持仓披露粒度（前十大 vs 全部持仓）已标注，静态归因的输入质量边界已明确
6. [ ] 所有计划进入一期的算法模块已评定为"可做"或"降级可做"
7. [ ] 首版核心 Schema 已定义并与实际字段对齐
8. [ ] 数据质量基线报告已输出，已知风险已列出对策
9. [ ] 至少有一个"暂不可做"的诚实判断（证明我们不是在回避坏消息）
10. [ ] 数据源风险登记册已产出，每个数据源的失效、限流和字段变化风险已记录

如果出现以下任一情况，则**一期范围需要调整**：

- 净值数据完全无法通过任何免费路径稳定获取
- 持仓数据字段覆盖率 <70%，导致静态归因无法稳定运行
- 风格因子指数日收益数据不可得，导致风格暴露分析无法做净值回归
- AKShare 关键接口长期不可用，且 A 级备选路径验证失败（两条都断了）

## 0.13 后续衔接

第零阶段的结果直接影响一期建设：

1. **字段映射** → 数据适配器代码中 `raw_column → canonical_name` 的转换逻辑，以及 `schemas.py` 中 Pydantic 模型的实际字段名
2. **覆盖率报告** → 一期页面"数据质量"页的展示内容和 `ResearchPacketMetadata.missing_fields` 的生成逻辑
3. **披露粒度** → 静态归因和持仓分析页面需要标注"基于前十大持仓"还是"基于全部持仓"
4. **算法评估** → 一期 Tool API 中哪些接口可以 `implemented: true`，哪些标注为实验性
5. **风险登记册** → 一期数据任务的重试策略、备用数据源切换逻辑、监控告警规则
6. **A 级路径验证** → 如果官方披露路径可用，一期优先使用；如果仅 AKShare 可用，一期在数据质量页标注数据源风险
